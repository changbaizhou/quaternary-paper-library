# 第四纪论文库综合升级设计

## 1. 目标

将现有的单机论文管理网页升级为适合第四纪地质学研究生长期使用的本地知识库。系统需要在约 10,000 篇论文规模下，稳定完成论文入库、自动识别、人工校对、检索、阅读、批注、知识整理和备份恢复。

本次升级采用混合增强路线：本地 PDF 解析和 OCR 是基础，公共学术元数据用于校验，Qwen 只处理经过限制的题名、摘要和前几页文本。所有自动识别结果必须经过人工确认后才能正式入库。

## 2. 已确认约束

- 运行方式：Windows 单机网页，继续支持双击批处理文件启动。
- 技术栈：保留 Node.js、Express、原生网页前端和 SQLite。
- 数据规模：目标支持 1,000 至 10,000 篇论文。
- 本地数据：PDF、完整正文、SQLite、个人笔记和批注只保存在本机。
- AI 输入：只允许发送题名、摘要和前 5 页中最多 20,000 个字符的提取文本。
- 翻译输入：现有翻译功能只发送用户明确选中的文本，并继续执行 6,000 字符上限；它不触发自动论文分析。
- 入库规则：AI 和联网识别只生成草稿，必须人工确认后入库。
- Git 身份：所有提交的作者和提交者均为 `changbaizhou <188980047+changbaizhou@users.noreply.github.com>`。
- 密钥安全：API Key 只保存在被 Git 忽略的本机配置文件中，不能写入数据库、日志、测试、文档或提交记录。

## 3. 总体架构

系统继续采用浏览器、Express 服务和 SQLite 三层结构。升级只在相关阶段拆分现有的大文件，不进行无关重写。

后端按职责拆分为以下模块：

- 论文仓储：论文、草稿、笔记、阅读进度和回收站操作。
- 数据库迁移：按版本升级结构，并保证旧数据可继续读取。
- 处理任务：记录上传、解析、OCR、联网补全、AI 分析和索引状态。
- 元数据识别：管理本地解析结果、公共元数据和 AI 候选值。
- 查重合并：判断文件、DOI 和题名重复，并安全合并关联数据。
- 检索索引：维护英文全文索引、中文 n-gram 索引和结构化筛选。
- 阅读标注：保存高亮、批注、引用摘录、译文和页面位置。
- 备份恢复：创建、校验、恢复数据库备份和完整资料包。

前端按工作区拆分为论文库、待确认、阅读器和维护设置，并把 API、状态管理和通用组件从 `public/app.js` 中逐步分离。

## 4. 分阶段交付

### 4.1 阶段一：数据基础

首先修复已入库论文无法保存的问题，并增加以下能力：

- 编辑并保存已入库论文的元数据、分类、阅读状态和研究笔记。
- 笔记空闲后自动保存，显示“保存中、已保存、保存失败”状态。
- 正式元数据使用明确的“保存更改”操作，避免无意覆盖。
- 论文删除先进入回收站，恢复时保留 PDF、笔记、批注和进度。
- 清空回收站时再次确认，随后才允许删除本地 PDF。
- 上传前计算 SHA-256 文件指纹。
- 按文件指纹、规范化 DOI、规范化题名与年份依次检查重复。
- 重复论文可以取消上传、保留两份或合并。
- 合并时保留非空元数据、所有笔记、阅读进度、批注和论文关系，并记录合并结果。
- 数据库升级、批量处理和合并前自动创建数据库备份。
- 提供手动数据库备份、完整资料包导出和恢复入口。

阶段一完成后，现有论文应当可以安全编辑、删除、恢复、查重和备份。

### 4.2 阶段二：识别引擎

上传任务依次经过以下步骤：

1. 保存文件并计算指纹。
2. 本地提取 PDF 文本和文件属性。
3. 文本不足时对前 5 页执行本地 OCR。
4. 从原文识别 DOI、题名、作者、期刊、年份、摘要和作者关键词。
5. 如果识别到 DOI，通过可替换的公共学术元数据提供器补全字段。
6. 如果没有 DOI 或字段存在缺失、冲突，再执行题名检索补全。
7. 将允许发送的有限文本交给 Qwen，返回结构化关键词、分类和研究卡建议。
8. 合并候选值，生成待确认草稿。

每个候选字段都记录值、来源、置信度、证据和获取时间。默认优先级为：

1. DOI 对应的公共注册元数据。
2. PDF 正文中明确出现的内容。
3. PDF 文件属性。
4. Qwen 的结构化推断。
5. 文件名兜底结果。

优先级不是无条件覆盖。当高优先级结果为空、明显截断或与正文强冲突时，待确认页面同时展示候选值，由用户选择。Qwen 必须返回符合服务端 JSON Schema 的结果；无法解析、缺少证据或超出允许分类词表的内容只作为低置信度建议。

Qwen 输入由服务端统一裁剪，最多包含题名、摘要和前 5 页中的 20,000 个字符。完整 PDF、其余正文、个人笔记、批注和数据库记录不得进入请求。

待确认页面允许单篇重试本地解析、联网补全或 AI 分析。已入库论文可以主动“重新识别”，但新结果先进入差异审阅，不自动覆盖现有内容。

### 4.3 阶段三：检索与知识库

检索采用 SQLite FTS5 和结构化索引，不在 Node.js 中读取全部论文后逐条筛选。

- 英文和数字内容使用 FTS5 `unicode61` 分词及 BM25 字段权重。
- 中文检索文本在本地生成双字 n-gram 词元，与英文索引一起查询。
- 两个汉字以下的短查询使用受字段索引约束的精确或前缀匹配。
- 中英文术语词典扩展第四纪地质学同义词，例如地名译名、时期名称、方法和指标别名。
- 结果按相关度、年份、创建时间、更新时间、题名和阅读状态排序。
- 筛选控件只展示数据库中真实存在的主题、区域、时期、材料、方法、指标、期刊和阅读状态。
- 列表使用服务端分页，默认每页 50 条。
- 用户可以保存、重命名和删除检索条件。
- 搜索结果显示命中字段和高亮片段。

相似论文使用可解释的本地评分：题名和摘要词元、作者关键词、主题、区域、时期、材料、方法及指标按权重计算。结果展示主要相似原因，不依赖外部向量数据库。

论文关系支持 `引用`、`被引用`、`同研究区`、`同方法`、`支持`、`冲突` 和 `自定义`。系统从本地参考文献文本中提取 DOI；只有被引用论文已经入库时才自动建立引用关系，其他关系由相似度建议或人工确认建立。

### 4.4 阶段四：阅读与知识整理

阅读器保留连续滚动、高清渲染、按可见区域懒加载、缩放、页码跳转、上次阅读位置和每篇论文一个主书签。

新增以下能力：

- 在当前 PDF 内搜索文字，并跳转到匹配位置。
- 使用缩略页导航长论文。
- 选中文字后显示翻译、高亮、添加批注和保存引用操作。
- 一篇论文可以有多个高亮、批注和引用摘录。
- 每条标注保存页码、选中文本、页内位置、颜色、批注、译文和创建时间。
- 点击标注可以重新打开论文并跳到对应页面和位置。
- 原文和译文可以一并保存为引用摘录。
- 标注、译文和笔记进入全文检索，但不会发送给 AI。

研究卡包含研究问题、研究区、材料与方法、年代框架、核心结论、局限性、可引用观点和个人备注。Qwen 生成的内容保存在独立的“AI 建议”区域，必须逐项采用，不能直接覆盖个人笔记。缺少证据的字段显示“原文信息不足”，不得编造。

### 4.5 阶段五：性能与维护

- 上传和识别任务具有 `queued`、`running`、`succeeded`、`failed` 和 `cancelled` 状态。
- 程序重启时，将未完成的 `running` 任务恢复为 `queued`。
- OCR 默认单任务运行，联网任务最多并发 2 个，防止电脑和 API 被占满。
- 每个步骤记录耗时、错误阶段和重试次数，不记录密钥或完整正文。
- 维护页显示任务队列、失败重试、批量重新识别、索引重建和重复扫描。
- 统计区域显示论文总数、待读/阅读中/已读数量、年度和分类分布，不作为首页宣传内容。
- 设置页只显示服务是否启用、模型名和连通状态，不回显完整 API Key。

## 5. 数据模型

在保留现有 `drafts` 和 `papers` 数据的基础上，增加版本化迁移和以下结构：

- `schema_migrations`：记录已执行的迁移版本。
- `processing_jobs`：保存处理类型、状态、步骤、进度、错误和重试信息。
- `paper_files`：保存路径、文件指纹、大小、页数和删除状态。
- `metadata_candidates`：保存字段候选值、来源、置信度和证据。
- `paper_analysis_suggestions`：保存尚未采用的 AI 研究卡和分类建议。
- `annotations`：保存高亮、批注、引用摘录、译文和页面定位。
- `paper_relations`：保存论文之间的有向关系、来源和确认状态。
- `saved_searches`：保存检索名称和结构化查询 JSON。
- `backup_records`：保存备份类型、路径、校验值、大小和创建时间。
- `papers_fts`：保存可检索文本和字段权重所需内容。

`papers` 增加规范化 DOI、规范化题名、删除时间和版本号。更新正式元数据时使用版本号防止过期页面覆盖新内容。所有迁移必须幂等，并在临时数据库和现有结构副本上通过测试。

## 6. 页面工作流

### 6.1 论文库

默认首页是论文库。顶部保留上传、全局搜索和任务状态；主体使用分页列表或紧凑表格，支持排序、筛选、批量选择、编辑、删除和相似论文入口。阅读状态和进度在列表中可见。

### 6.2 待确认

待确认工作区展示 PDF 首页预览、识别字段、候选来源、置信度和冲突。用户可以修改字段、切换候选值、重试某个处理阶段，最后确认入库。未确认草稿不得出现在正式检索结果中。

### 6.3 阅读器

阅读器占据主要空间，工具栏保持紧凑。文内搜索和缩略页位于可折叠侧栏，研究卡和标注列表位于另一可折叠侧栏。返回论文库时必须完全释放当前 PDF 状态，打开下一篇论文不受前一篇影响。

### 6.4 维护设置

维护设置包括回收站、重复论文、任务队列、备份恢复、索引维护、AI 与 OCR 状态。危险操作必须明确说明影响范围并二次确认。

## 7. 错误处理与恢复

- 上传保存成功后，即使 OCR、公共元数据或 Qwen 失败，也必须保留文件和可恢复草稿。
- 每个处理步骤独立重试，已经成功的步骤不重复调用。
- 联网请求设置超时、长度限制和有限重试；遇到限流时使用退避等待。
- 数据库写入使用事务，合并或恢复失败时整体回滚。
- 自动数据库备份滚动保留最近 30 份，用户可以在设置中调整保留数量。
- 完整资料包包含 SQLite、PDF、备份清单和不含密钥的配置说明。
- 恢复前校验清单和文件哈希，并先备份当前数据库。
- UI 错误信息说明失败阶段和可执行动作，不直接显示供应商返回的敏感内容。

## 8. 性能要求

- 10,000 篇论文的普通检索应在本机数据库查询阶段 500 毫秒内返回首屏结果。
- 论文列表只传输当前页和筛选汇总，不返回完整正文。
- PDF 页面按可见区域渲染，缩放时优先保留当前位置。
- 搜索索引在论文、笔记和标注变更后增量更新；批量操作完成后允许重建校验。
- 长任务不阻塞健康检查、论文检索和已入库 PDF 阅读。

## 9. 测试与验收

### 9.1 自动测试

- 单元测试：元数据提取、候选合并、置信度、查重、分类、中文词元和相似度评分。
- 数据库测试：迁移、编辑、回收站、恢复、合并、任务恢复、标注、关系、备份和 FTS 同步。
- API 测试：上传、确认、编辑、删除、分页检索、翻译、标注、任务重试和恢复。
- 浏览器测试：上传审核、保存编辑、搜索筛选、连续阅读、跳转、翻译、批注和返回后打开新论文。
- 视觉检查：桌面和窄屏下没有文字溢出、控件重叠、空白 PDF 或错误状态残留。
- 安全检查：Git 跟踪文件和提交差异中不存在 API Key、论文文件或 SQLite 数据。

### 9.2 阶段验收

每个阶段必须满足以下条件后才能提交和推送：

1. 新增行为具有对应测试。
2. 现有测试全部通过。
3. 对现有 SQLite 数据执行迁移演练并验证论文数量和关键字段不变。
4. 手工完成该阶段的主要用户流程。
5. 检查提交作者、工作区差异和敏感信息。

## 10. 明确不做的内容

- 不改造成云端多用户系统。
- 不上传完整 PDF 或完整正文到 Qwen 或其他 AI 服务。
- 不自动用 AI 内容覆盖人工确认的数据或个人笔记。
- 不在本阶段安装和维护本地大语言模型或独立向量数据库。
- 不实现论文全文批量翻译。
- 不提供跨设备实时同步；跨设备迁移使用完整资料包。

## 11. 实施原则

本文件是五个阶段的总体设计，不生成一个覆盖全部工作的超大实施计划。严格按“数据基础、识别引擎、检索知识库、阅读知识整理、性能维护”的顺序，为每个阶段分别编写实施计划、实现、验收、提交和推送；前一阶段完成后才为下一阶段展开详细计划。

每个阶段都要保持系统可启动、可回滚和可继续使用。所有代码、测试和文档由 `changbaizhou` 身份提交，阶段验收通过后推送到现有 GitHub 仓库。
